## 1. תקציר מנהלים

לום (Lume) היא ספקית חשמל בריטית קטנה המשרתת כ-12,000 משקי בית. ההוצאה הכי
כואבת שלה היא **הקנס שרשת החשמל גובה כשתחזית הצריכה החצי-שעתית שגויה** — קנס
שעד היום התגלה רק *למחרת*, מתוך קובץ שהרשת שולחת בדוא"ל, כשהכסף כבר אבד.

בנינו עבור לום **חדר בקרה תפעולי** שמציג את צריכת החשמל של כל הלקוחות כפי שהיא
נוחתת, כמעט בזמן אמת, ומזהיר את צוות המשמרת *לפני* שחלון חצי-השעה חורג מהסף.
לצד זה בנינו **משטח שאילתות לאנליסטים** המאפשר לתחקר את ההיסטוריה, ו**בסיס
תפעולי** שעולה בפקודה אחת ושורד הפעלות מחדש בלי לאבד נתונים.

המערכת בנויה כ**צינור נתונים זורם בן שבע שכבות**, כל שכבה עושה דבר אחד היטב
ומעבירה הלאה: שגשר כניסה מהיר (FastAPI) → תור הודעות עמיד (Kafka) → זיהוי
כפילויות (Redis) → מנוע עיבוד והזרמה (PySpark) → אחסון קבע (PostgreSQL) → מצב
חלון בזמן אמת (Redis) → דאשבורד (Grafana). במקביל רץ מתזמן תחזוקה (APScheduler).

**כל הדרישות הפונקציונליות בוצעו ואומתו בהרצה חיה:** לוח בקרה חי המתעדכן תוך
שניות, שש שאילתות ההגנה (כולן רצות), ובסיס תפעולי עמיד. במהלך הפרויקט פתרנו
למעלה מעשרה באגים אמיתיים — חלקם עקרוניים (זמן-קיר מול זמן-נתונים, אובדן נתונים
בכשל כתיבה, ביצועי כתיבה). נותרו מספר החלטות פתוחות וסיכונים שיוריים מודעים,
המפורטים בסעיף 13.

## 2. רקע עסקי — החברה והבעיה

### 2.1 החברה

לום היא ספקית חשמל בריטית מהדור החדש ("challenger retailer") המשרתת כ-12,000
משקי בית, רובם בצפון ומזרח לונדון, על תמהיל תעריפים: תעריף רגיל (Standard) ותעריף
מבוסס-שעות (Time-of-Use, ToU). היא קונה חשמל בשוק הסיטונאי ומוכרת אותו ללקוחות,
והרווח על כל יחידת אנרגיה (קוט"ש) דק מאוד. מכאן שכל הוצאה מיותרת פוגעת ישירות
ברווחיות.

### 2.2 הבעיה — הקנס החצי-שעתי

הרגולציה הבריטית מחייבת כל ספק לדווח מראש כמה חשמל הוא צופה שלקוחותיו יצרכו בכל
חלון של חצי שעה — **48 חלונות ביממה**. אם הצריכה בפועל חורגת מהתחזית, בכל כיוון,
הספק משלם **קנס לרשת**. זוהי ההוצאה הנשלטת הגדולה ביותר של לום.

הבעיה: עד היום הקנס התגלה רק *למחרת*, מתוך קובץ CSV שהרשת שולחת בדוא"ל — כלומר
אחרי שהכסף כבר אבד ואי-אפשר לעשות דבר. המנכ"ל הטכנולוגי (CTO) החליט שזה חייב
להשתנות.

### 2.3 ההזדמנות — מונים חכמים בזמן אמת

מה שהשתנה הוא שפריסת המונים החכמים הגיעה לכיסוי מספיק, וכעת ספק מונים חיצוני
יכול לשדר אלינו את קריאת הצריכה של כל בית **כל חצי שעה, כמעט בזמן אמת** — דחיפה
מעל webhook ברגע שהקריאה נאספת. אותו ספק גם חושף **ארכיון היסטורי** דרך REST
מבוסס-דפים, כך שניתן למלא פערים אחרי הפסקה, וכך שהאנליסטים יכולים לתחקר את אותו
מקור אמת שממנו מגיע הזרם החי.

**המנדט:** לבנות את חדר הבקרה סביב הפיד הזה. צוות המשמרת הלילי (שני אנשים
בסבב, ואנליסט תורן) צריך לראות את הצריכה כפי שהיא נוחתת ולקבל אזהרה *לפני* שחלון
חצי-השעה חוצה את סף הקנס. צוות האנליסטים צריך משטח SQL אמין מספיק כדי להריץ
שאילתות חיות בישיבת הבוקר ולהגן עליהן בביקורת הרבעונית של הרגולטור.

## 3. מקור הנתונים

ספק המונים חשוף דרך אפליקציה מכולתית (`meter-vendor`) הכלולה בפרויקט, ומציג שני
משטחים:

### 3.1 ארכיון היסטורי (Historical Archive)

REST מבוסס-דפים עם סמן (cursor). משמש לטעינה הראשונית ולמילוי פערים:

```http
GET /readings?since=2013-12-01T00:00:00Z&cursor=<opaque>&limit=1000
→ 200 { "readings": [ {delivery_id, meter_id, household_id, reading_time, kwh, received_at}, ... ],
        "next_cursor": "..." | null }
```

### 3.2 מנוי חי (Live Subscription)

דחיפת webhook. משמש לזרם החי:

```http
POST /subscriptions   { "webhook_url": "http://your-ingest:port/path" }
→ 200 { "subscription_id": "..." }

POST <your webhook_url>
     { "delivery_id": "...", "readings": [ {meter_id, household_id, reading_time, kwh, received_at}, ... ] }
→ 200   (כל תשובה אחרת = retry עם backoff; הסמן מתקדם רק על 200,
         כך שהחוזה הוא at-least-once וחובה לבצע dedup על delivery_id)
```

הספק מקדם שעון סימולציה במהירות רפליי שניתנת לכוונון, כך שהדאטה ההיסטורי מתנהג
כפיד חי. בברירת מחדל יום סימולציה אחד = 3 שניות קיר (`REPLAY_SECONDS_PER_DAY=3`,
≈5,500 קריאות לשנייה). מאיטים לדמו ומאיצים לבדיקות עומס.

### 3.3 נתוני עזר (Reference Data)

מצורפים לאותו volume שהספק קורא ממנו:

- **קובץ לקוחות** — שורה לכל משק בית, עם מקטע ה-ACORN, סוג התעריף (Std/ToU),
  גודל משק בית נומינלי, ותווית אזור-מיקוד יציבה. משמש להעשרת הקריאות למשטח
  האנליסטים ולפילוחי הדאשבורד.
- **פיד מזג אוויר שעתי** — טמפרטורה חיצונית וקוד מזג-אוויר גס בהיתרו (Heathrow),
  שעתי. משמש להסבר שונות בצריכה, בעיקר לשאלות גלי הקור.

### 3.4 מקור הדאטה הגולמי והטרנספורמציות

הדאטה מבוסס על מערך **London Datastore / Low Carbon London (LCL)** — קריאות
מונים חכמים אנונימיות. שלב העיבוד המקדים (`vendor/download_and_preprocess.py`)
מוריד כ-765MB, סורק מקור של ~167 מיליון שורות, ושומר רק את חלון הרפליי. תוך כדי
מבוצעות טרנספורמציות הופכות-נתונים-לשמישים:

- **סינון חלון** — נשמרות רק קריאות בטווח `[START − 30 ימים, END]`. המקור משתרע
  על נוב' 2011 – פבר' 2014; ברירת המחדל שומרת נוב' 2013 – פבר' 2014.
- **אזור זמן** — ה-`DateTime` במקור הוא שעון מקומי בריטי (BST/GMT); מומר דרך
  `Europe/London` ל-UTC כך שמעברי שעון-קיץ נוחתים בשעה הנכונה.
- **גיזום חריגים** — `kwh` מחוץ ל-`[0,50]` נזרק (קפיצות שליליות מאיפוסי מונה,
  ערכים גבוהים אבסורדיים משגיאות ייצוא סולארי).
- **כיסוי משקי בית** — משקי בית עם פחות מ-50% מהקריאות הצפויות בחלון נזרקים
  (בניסוי המקורי בתים הצטרפו ועזבו לאורך 27 חודשים).
- **אזור מיקוד** — מסונתז דטרמיניסטית מ-`household_id` כ-`N1-A` … `N1-H`. מפתח
  קיבוץ יציב, לא מיקוד אמיתי.
- **קבוצת ACORN** — מסונתזת דטרמיניסטית עם התפלגות בריטית מציאותית (30% A-E /
  30% F-J / 40% K-Q). ההתנהגות אינה מתואמת בפועל לתווית.
- **גודל משק בית** — נגזר מאות ה-ACORN לפי טבלת פרוקסי מפורסמת.
- **מזג אוויר** — תצפיות שעתיות מהיתרו ממאגר NOAA Global Hourly (ISD).

### 3.5 "כיעור" שהספק מדמה — והמערכת חייבת לעמוד בו

הספק מדמה על דרישה (`make vendor-chaos`) את הבעיות האמיתיות הבאות, והמערכת חייבת
להתמודד איתן **בלי התערבות מפעיל**:

- **חבילות כפולות** — אותו `delivery_id` יישלח לעיתים פעמיים. ה-dedup עלינו.
- **קריאות לא-מסודרות בתוך חבילה** — חותמות הזמן בתוך חבילה אינן בהכרח מונוטוניות
  (הספק אוסף מכמה ריכוזים אזוריים במקביל).
- **הפסקות** — הספק יפסיק לשדר לדקות. ההתראה "הפיד אבד" חייבת לקפוץ מהר, והגיבוי
  חייב להשלים בחן כשהשידור חוזר.
- **תיקונים מאוחרים** — קריאה מחבילה קודמת תיפלט מחדש לעיתים עם `kwh` מתוקן (ואותו
  `meter_id`+`reading_time`). משטח האנליסטים חייב לשקף את הערך **המתוקן**.

## 5. התמונה הגדולה — איך המערכת עובדת

אפשר לחשוב על המערכת כעל **קו ייצור** שמעביר כל קריאת מונה מהרגע שהיא נוחתת ועד
שהיא מופיעה על המסך ובבסיס הנתונים. בנינו אותו בשבע שכבות, וכל שכבה עושה דבר אחד
היטב ומעבירה הלאה.

ספק המונים שולח אלינו את הנתונים בשתי דרכים: **דחיפה חיה** (כל חבילה נשלחת מיד
ומצופה אישור תוך 10 שניות) ו**ארכיון היסטורי** (בקשה-תשובה מבוססת דפים, ששימשה
לטעינה הראשונית ולמילוי פערים אחרי הפסקות).

הזרם החי נכנס דרך **שגשר כניסה** — שירות קטן ומהיר שכל תפקידו לקבל את החבילה,
לחתום עליה בחותמת זמן, ולהחזיר אישור *מיד*, לפני שנגע בתוכן. למה זה חשוב? כי
עיבוד הנתונים (בדיקה, ניקוי, כתיבה לדיסק) לוקח זמן לא-צפוי, ואם נחכה לו לפני
שנאשר — הספק יחשוב שלא קיבלנו וישלח שוב, וניצור כפילויות. לכן השגשר רק "מקבל
ומאשר", ושופך את החבילה לתור הודעות.

**תור ההודעות** הוא חיץ שמנתק בין קצב הקבלה לקצב העיבוד, ושומר את הנתונים לדיסק
כך שגם אם מנוע העיבוד נופל — שום קריאה לא הולכת לאיבוד. חילקנו אותו לשני נתיבים:
אחד לזרם החי ואחד לגיבוי ההיסטורי, כדי שמשיכה גדולה של היסטוריה לא תחנוק את
הנתונים החיים.

**מנוע העיבוד** הוא הלב. כל כמה שניות הוא לוקח אצווה של חבילות ועובר עליהן בשמונה
צעדים: בודק שאינן כפילות, מפרק כל חבילה לקריאות בודדות, מאמת שכל קריאה הגיונית,
כותב את הקריאות הנקיות לבסיס הנתונים בכתיבה "מותנית" שמחליפה ערך רק אם הוא חדש
יותר, ומעדכן את מצב החלון החי בזיכרון המהיר.

בקצה יש שני מקומות שבהם הנתונים יושבים: **בסיס נתונים** קבוע (מקור האמת
לאנליסטים ולהיסטוריה) ו**זיכרון מהיר** (מצב החלון הנוכחי, מוכן לתצוגה תוך שניות).
לוח הבקרה קורא משניהם — את המספרים החיים מהזיכרון המהיר, ואת ההיסטוריה והמגמות
מבסיס הנתונים. במקביל רץ **מתזמן** שמבצע משימות תחזוקה תקופתיות.

**זרימת הנתונים בקצרה:**

```
                                    ┌──────────── partition 0 (חי) ────────────┐
ספק → webhook → [1] שגשר כניסה ──────┤                                          │
                (FastAPI)            └──→ [2] Kafka readings-raw ──→ [4] Spark ──┤
ספק → REST archive → [2'] גיבוי ─────┐                              (8 צעדים)   │
                     (Python)        └──── partition 1 (גיבוי) ─────┘           │
                                                                                │
        [3] Redis dedup ←──────────────────────────────────────────────────────┘
                                            │
                    ┌───────────────────────┼───────────────────────┐
                    ▼                        ▼                       ▼
            [5] PostgreSQL          [6] Redis window:*        rejected/dups
            (מקור אמת)              (מצב חלון חי)             (איכות)
                    │                        │
                    └────────→ [7] Grafana ←─┘   ⟸ [מתזמן] APScheduler
                                (refresh 5s)
```

## 6. שבע השכבות — פירוט מילולי מלא

לכל שכבה: **מה היא עושה**, **למה כך**, **כיצד היא עמידה**, ו**מקרי הקיצון**
שהיא מטפלת בהם.

### שכבה 1 — שגשר הכניסה (FastAPI) · `ingest/app/main.py`

**מה.** מקבל את דחיפת ה-webhook מהספק ומחזיר אישור (HTTP 200) תוך עשיריות
שנייה, לפני כל עיבוד כבד. הוא חותם על החבילה בזמן-קיר (`ingest_time`) ודוחף אותה
לטופיק `readings-raw` ב-Kafka, **partition 0** (נתיב חי).

**למה.** בחרנו שירות אסינכרוני (FastAPI/asyncio) כדי שיוכל לטפל בהרבה בקשות
במקביל בלי להיתקע. תפקידו היחיד הוא "לקבל ולאשר": הוא **לא** מאמת ערכי kwh, **לא**
מזהה כפילויות, ו**לא** כותב לבסיס נתונים. ה-200 מוחזר ברגע שההודעה נכנסת ל-buffer
של ה-producer — לפני סבב הרשת ל-Kafka — מה שמספק את דרישת האישור תוך 10 שניות
במרווח עצום ושומר את השגשר חסר-מצב (stateless).

**עמידות.** השירות מדווח על עצמו כ"לא בריא" (HTTP 503 ב-`/healthz`) אם ה-producer
של Kafka לא עלה, או אם ההרשמה לספק נכשלה אחרי כל הניסיונות החוזרים — כך
ש-Docker יפעיל אותו מחדש לבד במקום "להירדם בריא". הגדרות ה-producer:
`acks=1` (אישור מנהיג), `linger_ms=5` (מיקרו-אצווה), `compression_type=lz4`,
`retries=5`.

**הרשמה אידמפוטנטית.** הספק שומר את רשימת המנויים לדיסק ומשחזר אותם בכל הפעלה
מחדש; לכן לפני שהשגשר נרשם, הוא **מנקה תחילה כל מנוי קודם שמצביע על עצמו**
(`GET /stats` → `DELETE` לכל מנוי עם אותו `webhook_url`) ורק אז נרשם מחדש. כך כל
הפעלה מחדש של השגשר או של הספק מותירה **מנוי יחיד** לשגשר, במקום להצטבר למספר
מנויים שכל אחד מהם מקבל עותק של כל פרוסה — דבר שהיה מכפיל את העומס החי ודוחף את
מנוע העיבוד מאחור ללא צורך (ראה סעיף 10.2).

> *הערה פנימית:* השגשר אינו מוצג בגרפנה.

### שכבה 2 — תור ההודעות (Kafka) + שירות הגיבוי

**מה.** Kafka הוא המאגר העמיד שמפריד בין מי שמקבל (השגשר) למי שמעבד (Spark), כך
שכל צד עובד בקצב שלו. שלושה תפקידים: לחצוץ בין הקצבים, לאפשר "לחזור אחורה" ולקרוא
שוב את אותן הודעות (replay) אחרי כשל, ולהבטיח שאם מנוע העיבוד נופל — שום קריאה לא
הולכת לאיבוד.

**גם ב-RAM וגם בדיסק — ובכוונה.** Kafka כותב כל הודעה ל**יומן append-only על
הדיסק** (קובץ לוג רציף לכל partition, ב-volume בשם `kafka-data`) — וזה מה שנותן
את העמידות: גם אם כל הקונטיינרים ייפלו, ההודעות של השעות האחרונות עדיין שם
והעיבוד ימשיך בדיוק מהנקודה שבה עצר. במקביל, הקריאות הטריות מוגשות מ**זיכרון ה-page
cache של מערכת ההפעלה (RAM)** — כך ש-Spark קורא הודעות חדשות ישר מה-RAM בלי לגעת
בדיסק, וזה מה שנותן את המהירות. בקיצור: **הדיסק = עמידות, ה-RAM = ביצועים**.
שתי נקודות עדינות: (א) בחרנו `acks=1` עם replica יחיד — הברוקר מאשר ברגע שההודעה
ביומן המקומי וההצפה הפיזית לדיסק אסינכרונית; פשרת מהירות-מול-עמידות סבירה לדמו על
לפטופ. (ב) גם השגשר מחזיק את ההודעה רגע קצר ב-buffer ב-RAM של ה-producer לפני
ש-thread ברקע שולח אותה — ולכן הוא יכול לאשר לספק תוך מילישניות.

**שני נתיבים ושמירת זמן.** חילקנו את הטופיק `readings-raw` לשני partitions:
partition 0 לזרם החי, partition 1 לגיבוי ההיסטורי — כך שמשיכה גדולה של היסטוריה
לא תחנוק את הנתונים החיים, כי לכל partition יש offset עצמאי שמתקדם בקצב שלו.
ה-`retention` מוגדר ל-**6 שעות** (`KAFKA_CFG_LOG_RETENTION_HOURS: 6` +
`retention.ms=21600000` בטופיק) — הוא חייב לעלות על זמן ההפסקה הגרוע ביותר של מנוע
העיבוד, כי נתונים שקיימים **רק** בזרם החי (תיקונים מאוחרים) אינם בארכיון, ולכן אם
Spark למטה זמן רב מה-retention הם נמחקים מהתור והגיבוי לא יכול לשחזר אותם (ראה
סעיף 10 ו-13). 6 שעות הן מרווח רחב כי Spark קם תוך שניות, ועלות הדיסק בקצב המתון
המומלץ (60ש'/יום) היא כ-12GB.

**שירות הגיבוי ההיסטורי (Python) · `backfill/app/main.py`.** מושך מהארכיון דף אחרי
דף ופרסם ל-partition 1, שומר "סימן מים" (`watermark.txt` — עד היכן הגענו), וכך
אחרי הפסקה הוא יודע להמשיך בדיוק מהמקום הנכון ולמלא את הפער. שני תפקידים: (1)
טעינה היסטורית מלאה בעלייה הראשונה, (2) מילוי פערים אוטומטי אחרי הפסקת פיד.
**תיקון שביצענו:** דפי גיבוי מקבלים `delivery_id` יציב מ**גיבוב-תוכן**
(`_content_hash`, prefix `bf_`) — הארכיון מספק `arch_<uuid>` אקראי בכל משיכה, ובלי
התיקון משיכה חוזרת אחרי קריסה הייתה חומקת מ-Redis; כעת אותו דף ממופה לאותו מזהה
ו-Redis מדלג עליו, ומחזיר את האידמפוטנטיות.

### שכבה 3 — זיהוי כפילויות (Redis) · `spark/jobs/dedup.py`

**מה.** לפני העיבוד, כל חבילה נבדקת מול הזיכרון המהיר בפעולה אטומית אחת:
`SET dedup:<delivery_id> 1 NX EX 86400` — "רשום את מזהה החבילה רק אם לא קיים".
אם הוא כבר קיים — זו כפילות, מדלגים, וכותבים שורה ל-`batch_duplicates`.

**למה.** בדיקה כזו בבסיס הנתונים, בקצב של עשרות אלפי קריאות בשנייה, הייתה
בלתי אפשרית; הזיכרון המהיר עונה במיקרו-שניות. השימוש ב-`SET NX EX` כפקודה **אחת**
(ולא SET ואז EXPIRE נפרדים) הוא קריטי: לולא כך, קריסה בין שתי הפקודות הייתה משאירה
מפתח בלי TTL שמדליף זיכרון לנצח.

**היקף.** Redis תופס רק את **אותה מעטפת** (אותו `delivery_id`) שנשלחה פעמיים —
בעיקר כפילות חיה. כפילות **בין** נתיב חי להיסטורי (מזהים שונים) נפתרת בכתיבה
המותנית ל-DB לפי `received_at` (שכבה 5).

**עמידות / fallback.** אם Redis אינו זמין — כל החבילות עוברות עם אזהרה, והכתיבה
המותנית בשכבה 5 סופגת כפילויות אמיתיות ברמת ה-DB (לאט יותר אך נכון). בנוסף,
הפונקציה `unmark_payloads` משחררת את סימוני ה-dedup כשכתיבה במורד הזרם נכשלת — כך
שריצה חוזרת תעבד מחדש את הקריאות במקום להפיל אותן בשקט (ראה שכבה 4).

### שכבה 4 — מנוע העיבוד (PySpark Structured Streaming) · `spark/jobs/process.py`

**מה.** קורא **חבילות** משני נתיבי התור (2 ה-partitions: 0=חי, 1=גיבוי), כאשר
לכל partition offset עצמאי, ומנהל את המיקום על הדיסק (checkpoint) כך שאחרי הפעלה
מחדש כל partition ממשיך בדיוק מנקודתו. שני הנתיבים נמשכים ל-**micro-batch אחד
מאוחד** (כל ~5 שניות): ההנדלר מפרק כל חבילה לקריאות בודדות, ואז הניקוי, הולידציה
והכתיבה רצים על כל האצווה ביחד.

**כתיבה מאוחדת.** הכתיבה ל-PostgreSQL **אינה מקבילה ואינה נפרדת** בין שני
הנתיבים — היא נעשית ברצף על **חיבור יחיד** (`foreachBatch` רץ ב-driver), עם
COPY+merge אחד לכל האצווה. כך אין תחרות בין הנתיבים על אותה שורה, וה-`DISTINCT ON
(meter_id, reading_time)` מבצע dedup גם **חוצה-נתיבים** באותו merge. ברמת Kafka
המחיצות עצמאיות; ברמת ה-DB הן מתנקזות לכתיבה אחת מאוחדת.

**הגבלת גודל אצווה.** `maxOffsetsPerTrigger=500` סופר **הודעות Kafka**, לא קריאות.
דחיפה חיה היא קומץ קריאות, אבל דף גיבוי נושא עד 1,000 — כך שתקרה גבוהה הייתה
מושכת מיליוני קריאות ל-driver בבת אחת → OutOfMemoryError ואצווה רב-דקתית שמקפיאה
את הדאשבורד. 500 שומר כל trigger קטן מספיק כדי שה-COPY יסיים בכמה שניות.

**תהליך העיבוד ב-8 צעדים — הגנות ומקרי קיצון:**

1. **דה-סריאליזציה** — כל הודעת Kafka (חבילה) מפוענחת מ-JSON. *הגנה:* הודעה
   פגומה נרשמת ל-log ומדולגת, לא מפילה את האצווה.
2. **דדופ (Redis `SET NX EX`)** — כל `delivery_id` נרשם אטומית; אם קיים → כפילות,
   מדולג ונכתב ל-`batch_duplicates`. *מקרה קיצון:* Redis לא זמין → כל החבילות
   עוברות עם אזהרה, והכתיבה המותנית בשכבה 5 סופגת כפילויות אמיתיות.
3. **Flatten** — כל חבילה מפורקת לקריאות בודדות. כאן חבילת גיבוי של 1,000 קריאות
   הופכת ל-1,000 שורות.
4. **ולידציה** — לכל קריאה: זמן ניתן-לפענוח, `kwh` קיים ובטווח `[0,50]`. *מקרה
   קיצון:* זמן **לא-תקין** (לא רק ריק) נדחה כראוי; קריאה שנכשלת נכתבת
   ל-`rejected_readings` עם הסיבה, לא נזרקת בשקט.
5. **כתיבה מותנית (COPY → temp → MERGE) לטבלה המחולקת** — **הכתיבה הקריטית היחידה**.
   הניתוב ל-partition לפי `reading_time` אוטומטי, ו-`synchronous_commit=off` על
   הסשן מסיר את עלות ה-fsync. *הגנה:* בכשל → `rollback` + שחרור סימוני הדדופ
   (`unmark_payloads`) + זריקת שגיאה, כך ש-Spark **לא** מקבע את ה-offset והאצווה
   תעובד שוב (אידמפוטנטי). *מקרי קיצון שנפתרים במנגנון אחד:* כפילות (אותו
   `received_at` וערך זהה → לא מחליף), תיקון מאוחר (`received_at` מאוחר → מחליף),
   **תיקון בו-זמני** (אותו `received_at`, ערך שונה, ingest חדש → דורס — ד.1),
   retry ישן (`received_at` מוקדם → לא מחליף), וסדר-לא-מונוטוני באצווה
   (`DISTINCT ON` בורר את החדש לפני הכתיבה).
6. **כתיבת דחיות** — append-only ל-`rejected_readings`. *הגנה:* רץ רק אחרי שהכתיבה
   הקריטית הצליחה (כדי שלא ירוץ פעמיים ב-retry); כשל כאן הוא best-effort עם rollback.
7. **עדכון חלון חי (Redis) — קדימה בלבד.** רק קריאות בחלון הנתונים הנוכחי
   (`max(reading_time)` באצווה). *מקרי קיצון:* קריאות גיבוי מחוץ לחלון מסוננות כדי
   לא לקלקל את הסכום החי; ואצווה **טהורת-גיבוי** שחלונה ישן מהחלון השמור **מדולגת**
   (החלון מתגלגל קדימה בלבד) כדי שלא ייסוג אחורה — ממצא ד.6.
8. **קיבוע offset** — מרומז דרך ה-checkpoint, רק אחרי שכל הצעדים הצליחו.

**הולידציה (Python) · `spark/jobs/validate.py`.** שלושה סוגי דחייה מובחנים:
`invalid_reading_time` (זמן חסר/לא-ניתן-לפענוח), `missing_kwh` (kwh חסר/לא-מספרי),
ו-`kwh_out_of_range` (מחוץ ל-`[0,50]` או NaN).

### שכבה 5 — האחסון (PostgreSQL) · `spark/jobs/db_writer.py` + `db/migrations/001_init.sql`

**מה.** הקריאות הנקיות נכתבות בכתיבה מותנית: מחליפים ערך קיים אם הגרסה החדשה
התקבלה מאוחר יותר, **או** באותו `received_at` כשהיא ingest חדש יותר עם `kwh` שונה
(תיקון בו-זמני — ד.1). כלומר התנאי הוא `EXCLUDED.received_at > clean_readings.received_at`
`OR (received_at זהה AND ingest חדש AND kwh שונה)`.

**מנגנון אחד שפותר את כל מקרי הקצה** — כפילויות (אותו `received_at` וערך זהה → לא
מחליף), תיקונים מאוחרים (זמן מאוחר → מחליף), ניסיונות חוזרים ישנים (זמן מוקדם →
לא מחליף), **ותיקון בו-זמני** (אותו `received_at`, ערך שונה, ingest חדש → דורס —
ממצא ד.1).

**חלוקה (partitioning).** `clean_readings` מחולקת לפי `reading_time` (חודשי +
DEFAULT), כך שכל כתיבה נוגעת רק ב-partition הנוכחי — הקטנת עלות תחזוקת האינדקסים
והאצת שאילתות הטווח (סעיף 12.1). הניתוב ל-partition אוטומטי, וה-`ON CONFLICT` עובד
כי מפתח החלוקה כלול ב-PK.

**הביצועים — בכמות.** הכתיבה אינה שורה-שורה אלא **COPY לטבלת ביניים זמנית
(`_stage_clean`) ואז MERGE יחיד מבוסס-קבוצה**. זה מהיר בעשרות מונים מ-`executemany`
בנפחים האלה (מאות אלפי קריאות לאצווה): COPY מזרים את השורות במעבר אחד, וה-INSERT…
SELECT…ON CONFLICT מחיל אותן בפקודה אחת. ה-`DISTINCT ON` בתוך ה-merge גם משטח
קריאות לא-מסודרות/כפולות בתוך האצווה לפני שהן נוגעות ב-`clean_readings`. בנוסף
מוגדר `work_mem='256MB'` כדי שמיון ה-DISTINCT ON יישאר בזיכרון.

**טריגר תיעוד תיקונים.** טריגר על `clean_readings` מתעד אוטומטית כל שינוי `kwh`
(ערך ישן, חדש, הפרש) ל-`meter_corrections_log` — ללא קוד נוסף.

### שכבה 6 — מצב החלון בזמן אמת (Redis) · `spark/jobs/redis_state.py`

**מה.** הזיכרון המהיר מחזיק ערכים מצטברים מוכנים לתצוגה, כך שהלוח מתעדכן תוך
שניות בלי לסכום מיליוני שורות בכל רענון. שמונה מפתחות פעילים:

| מפתח | תוכן |
|---|---|
| `window:start` | חותמת ISO של חלון ההתיישבות הנוכחי (30 דק') |
| `window:kwh` | סכום קוט"ש מתגלגל (`INCRBYFLOAT`, אטומי) |
| `window:meters_hll` | HyperLogLog של מזהי מונים שדיווחו (≈1% שגיאה, 12KB קבוע) |
| `window:meters` | מראה רגילה של `PFCOUNT` כדי שגרפנה תקרא ב-GET |
| `window:last_batch` | חותמת האצווה האחרונה (**TTL 150ש'**); כשפג → "feed dead" |
| `window:forecast` | תחזית לסוף החלון (מחושבת מחדש בכל אצווה) |
| `window:backfill_active` | נוכח (TTL 60ש') בזמן עיבוד גיבוי; באנר "ממלא פערים" |
| `stats:dups:5m` / `stats:invalid:5m` | מוני 5 דק' מתגלגלים לכפילויות/דחיות |

**תחזית מבוססת-כיסוי.** תחזית מבוססת-זמן (kwh ÷ דקות-שחלפו × 30) **אינה עובדת**
כאן: כל קריאה נחתמת על גבול חצי השעה, כך שכל הקריאות בחלון חולקות אותה חותמת זמן
ו"הדקות שחלפו" תמיד 0. במקום זה אנו מטילים מהכיסוי: לוקחים את הצריכה שנצברה עד כה
ומחלקים בשיעור המונים שכבר דיווחו. כשדיווחו 40% — התחזית פי 2.5 מהצריכה הנוכחית;
כשכולם דיווחו — התחזית שווה לסכום בפועל. המכנה (סך המונים בספר) נקרא פעם אחת
מטבלת `homes` בעליית Spark.

**מפתח `window:meters`.** תוסף ה-redis-datasource של גרפנה אינו מחזיר data frame
לפקודת `PFCOUNT`, אך כן ל-`GET` — לכן המנוע משקף את ספירת ה-HLL למפתח מחרוזת רגיל
שהפאנל "Active Meters" קורא.

**מחזור חלון — קדימה בלבד.** כשהאצווה שייכת לחלון 30-דק' **מאוחר יותר** מהשמור, כל
מפתחות `window:*` נמחקים והחלון מתחיל מאפס. אצווה שחלונה **ישן** מהשמור (טהורת-גיבוי)
**מדולגת** ואינה מגלגלת את החלון אחורה — כך הסכום החי לא נסוג זמנית (ממצא ד.6).

### שכבה 7 — הדאשבורד (Grafana) + המתזמן

**הדאשבורד (Grafana) · `grafana/dashboards/lume-operational.json`.** מסך אחד —
"Lume — Control Room" — עם מדדים חיים מהזיכרון המהיר (Redis) וגרפים היסטוריים
מבסיס הנתונים (PostgreSQL), שמתרענן כל **5 שניות**. פירוט מלא של 12 הפאנלים בסעיף 9.

**המתזמן (APScheduler) · `scheduler/app/main.py`.** רץ בצד ומבצע שלוש משימות
תחזוקה תקופתיות, **מעוגנות ב-זמן-הנתונים** ולא בשעון-הקיר:

1. **כל 30 דק' — בדיקת מונים שותקים.** מונים עם קריאות ב-30 הימים האחרונים שלא
   דיווחו ב-2 השעות האחרונות. שורה לכל מונה ל-`meter_silence_log`.
2. **יומי 03:00 — סיכום מונים יתומים.** מזהי מונים שמופיעים ב-`clean_readings`
   אך לא ב-`homes`. Upsert ל-`orphan_meters_summary`.
3. **יומי 04:00 — ארכוב וניקוי דחיות.** גלגול `rejected_readings` (לפי שעה + סוג)
   ל-`rejection_hourly_summary` הקבוע ב**סכימה מצטברת מדויקת** (`count = existing +
   new`), תוך סימון כל שורה שנקלטה ב-`archived=TRUE` כך שהיא נספרת בדיוק פעם אחת
   (ממצא ד.4), ואז מחיקת שורות בנות 7+ ימים.

כל השאילתות במתזמן עברו ל**פרמטרים מבונדים** (לא string-format — ממצא ד.3).
המשימות המזינות פאנלים רצות **מיד בעלייה** (במקום להמתין למחזור הראשון), עם
`misfire_grace_time=600` ו-`coalesce=True` כדי שריצה שאיחרה לא תידלג בשקט.

## 7. מבנה הנתונים — הטבלאות בבסיס הנתונים

**11 טבלאות קבע + טבלת-ביניים זמנית אחת** (`_stage_clean`, נוצרת בכל אצווה ונמחקת
ב-commit). מספרי השורות הם צילום חי במצב דמו (`make run`).

### ליבת הקריאות

```
clean_readings — ~21.5M שורות — לב המערכת, מקור האמת לקריאות הנקיות
  PARTITION BY RANGE (reading_time) — partitions חודשיים (2013-10..2014-03) + DEFAULT
  PK (meter_id, reading_time) · index: reading_time (local לכל partition)
  meter_id      varchar      NOT NULL
  reading_time  timestamptz  NOT NULL  ← מפתח החלוקה; ניתוב לכל partition אוטומטי
  household_id  varchar
  kwh           double       NOT NULL
  received_at   timestamptz            ← מכריע "מי חדש יותר" בכתיבה המותנית
  ingested_at   timestamptz  NOT NULL  ← טייברייקר כש-received_at זהה (ד.1)
```

### טבלאות ממד (לאנליסטים, נטענות פעם אחת ע"י load-dimensions)

```
homes — ~5,083 שורות — מונה ↔ משק בית, מקטע ACORN, תעריף, אזור מיקוד
  PK household_id · UNIQUE index meter_id  ← כל JOIN עובר דרך meter_id
  household_id, meter_id (NOT NULL), postcode_area,
  household_size, acorn_group, tariff_type (מנורמל ל-'ToU'/'Std')

weather_hourly — ~17,509 שורות — טמפ' שעתית בהיתרו (לשאלת גל הקור)
  PK observed_at (מעוגל לשעה)
  station (default 'EGLL-Heathrow'), temp_c,
  report_type (מעדיף 'FM-12' כשיש כמה דיווחים בשעה)
```

### תפעוליות ואיכות (המנוע כותב)

```
batch_log — שורה לכל מסירה שהתקבלה (בריאות הפיד, פאנלים A2/A3)
  PK delivery_id · source, received_at, reading_count, status, logged_at

batch_duplicates — שורה לכל חבילה כפולה שדדופ דחה
  PK delivery_id · detected_at, source, reading_count

rejected_readings — קריאות שנדחו בולידציה (חלון 7 ימים)
  PK id · index: received_at · index: (meter_id, rejection_type) · partial index WHERE NOT archived
  + delivery_id, meter_id, reading_time, kwh, received_at,
    rejection_type NOT NULL, rejection_detail, source, rejected_at,
    archived boolean NOT NULL DEFAULT false  ← סומן אחרי ארכוב מצטבר (ד.4)

meter_corrections_log — תיעוד אוטומטי של כל תיקון kwh (טריגר)
  PK id · index: (meter_id, reading_time)
  + old_kwh, new_kwh NOT NULL, delta (GENERATED), corrected_at
```

### אנליטיות (המתזמן כותב)

```
meter_lag_stats — עיכוב שידור לכל מונה (EWMA)
  PK meter_id · avg_lag_sec, p95_lag_sec, max_lag_sec, updated_at

meter_silence_log — מונים ששתקו 2+ שעות (כל 30 דק')
  PK id · meter_id, postcode_area, last_seen, silent_hours, detected_at

orphan_meters_summary — מונים בקריאות אך לא ב-homes (יומי)
  PK meter_id · first_seen, last_seen, reading_count, total_kwh

rejection_hourly_summary — סיכום שעתי קבוע של דחיות (נשמר לעד)
  PK (ts_hour, rejection_reason) · count
```

### בניית טבלאות האנליסטים ובעיות נתונים שנפתרו

- **`clean_readings`** נבנית **תוך כדי הזרמה** (לא ETL חד-פעמי): כל קריאה עוברת
  ולידציה ונכתבת בכתיבה מותנית על המפתח `(meter_id, reading_time)`. *בעיות שנפתרו:*
  כפילויות (לא נספרות פעמיים), קריאות לא-מסודרות באצווה (`DISTINCT ON`), תיקונים
  מאוחרים (דריסה + טריגר תיעוד), וזמנים פגומים (נדחים במקום להישמט בשקט).
- **`homes`** נטענת מ-`households.parquet`. *בעיה שנפתרה:* ה-JOIN חייב לעבור דרך
  `meter_id` (לא `household_id`) — הוספנו **index ייחודי** על `meter_id`. בנוסף
  `tariff_type` מנורמל ל-`ToU`/`Std` כבר בטעינה.
- **`weather_hourly`** נטענת מ-`weather.parquet`. *בעיות שנפתרו:* (א) לפעמים יש
  **כמה דיווחים לאותה שעה** — בחרנו את דיווח `FM-12` (הסדיר), ועיגלנו `observed_at`
  לשעה כמפתח; (ב) עיגון ל-UTC כדי שה-JOIN `date_trunc('hour', reading_time) =
  observed_at` ייפול נכון.

## 8. משטח האנליסטים — שש שאילתות ההגנה

הקובץ `queries/defence_queries.sql` מכיל את שש השאילתות, כתובות מראש ומורצות חי
בהגנה. כולן מעוגנות ל-`AT TIME ZONE 'UTC'` כדי להגן מפני שינויי אזור-זמן.

| # | שאלה | מה היא בודקת |
|---|---|---|
| 1 | צריכה כוללת בכל חלון 30-דק', ערב 2014-01-13 (17:00–22:00) | חישוב חלון, GROUP BY זמן, שלמות נתונים |
| 2 | 10 הבתים הצורכים ביותר ב-2014-01-13 + אזורי מיקוד | JOIN דרך `meter_id`, ACORN, תעריף |
| 3 | ממוצע יומי לפי קבוצת ACORN ותעריף | GROUP BY כפול, AVG על צבירה יומית (תת-שאילתה) |
| 4 | כיסוי מונים יומי (מונים ששתקו) | COUNT(DISTINCT), אחוז כיסוי, scalar subquery |
| 5 | אחוז צריכת שפל (00:00–06:00) ללקוחות ToU | SUM מותנה (CASE), EXTRACT(HOUR), NULLIF |
| 6 | קשר טמפ' היתרו ↔ צריכה יומית (גל הקור) | JOIN ל-weather, חלון OVER(), CORR() |

**בעיות שנפתרו בשאילתות עצמן:** שאלה 5 נכשלה כי `ROUND` אינו תומך ב-`double
precision` ללא המרה — נוסף cast מפורש ל-`numeric`. כל השאילתות עוגנו ל-UTC כדי
להגן מפני שינויי אזור-זמן.

## 9. חדר הבקרה — הדאשבורד בגרפנה

הדאשבורד **"Lume — Control Room"** הוא מסך אחד עם **12 פאנלים**, refresh כל 5
שניות, מקורות Redis (חי) ו-PostgreSQL (היסטוריה). להלן כל פאנל, מקור הנתונים,
והדרישה שהוא מממש.

### 9.1 פאנלים המממשים את דרישות ה-BRIEF

| # | מה המסך נדרש לענות | פאנל | מקור נתונים |
|---|---|---|---|
| A1 | האם פיד הספק בריא כרגע? | **Feed Status** (LIVE/DEAD) | Redis — `TTL window:last_batch` |
| A2 | מתי התקבלה המסירה האחרונה? | **Last Delivery Accepted** | PostgreSQL — `MAX(logged_at)` מ-`batch_log` |
| A3 | כמה מסירות ב-5 הדק' מול בסיס השעה? | **Deliveries — 5 min vs Baseline** | PostgreSQL — `batch_log` (5 דק' מול `count/12`) |
| B4 | סך קוט"ש החלון הנוכחי מול סף הקנס | **kWh — Current 30-min Window** (gauge) | Redis — `window:kwh` |
| B5 | נקודת חריגה חד-משמעית למפעיל עייף | אותו gauge — ספים ירוק→צהוב (9,720)→אדום (10,800) | Redis — `window:kwh` |
| C6 | תחזית לסוף החלון (אזהרה **לפני** חריגה) | **Forecast — End of Window** | Redis — `window:forecast` (מבוסס-כיסוי) |
| D7 | עדכון תוך שניות, בלי רענון ידני | כל הדאשבורד | `refresh: 5s` + מקורות Redis |

### 9.2 פאנלים נוספים (מעבר לדרישה)

| פאנל | מה הוא מציג | מקור |
|---|---|---|
| **Filling Gaps (Backfill)** | באנר "ממלא פערים" בזמן עיבוד גיבוי | Redis — `window:backfill_active` |
| **Active Meters** | כמה מונים דיווחו בחלון הנוכחי | Redis — `window:meters` |
| **Duplicates (5 min)** | מונה כפילויות מתגלגל | Redis — `stats:dups:5m` |
| **Invalid Readings (5 min)** | מונה קריאות פסולות מתגלגל | Redis — `stats:invalid:5m` |
| **kWh per 30-min Settlement Window — Last 24 h** | היסטוריית צריכה לפי חלון | PostgreSQL — `clean_readings` |
| **Rejection Rate by Type — Last 24 h** | שיעור דחיות לפי סוג | PostgreSQL — `rejected_readings` |
| **Silent Meters — Last Detection** | מונים ששתקו בזיהוי האחרון | PostgreSQL — `meter_silence_log` |

### 9.3 ליטוש הדאשבורד (בעיות תצוגה שנפתרו)

- ספי הצבע ב-gauge כוילו לגודל האמיתי של מצבת המונים.
- פאנלים של מוני 5 דקות מציגים "0" במקום "אין נתונים" כשאין פעילות.
- "Active Meters" קורא ממפתח `window:meters` רגיל (לא PFCOUNT) שגרפנה כן מציגה.
- "Feed Status": ה-TTL של `window:last_batch` הוחזר ל-150 כדי שיתאים למיפוי הלוח
  (1–150 → LIVE) וש-"DEAD" יקפוץ תוך ~150ש' (ראה סעיף 10).

## 10. הבעיות שפגשנו וכיצד פתרנו

לאורך הדרך עברנו על הקוד שורה-שורה, הרצנו את המערכת בפועל, וזה חשף שורה של בעיות
אמיתיות שתיקנו ואימתנו בהרצה חיה.

### 10.1 בעיות עקרוניות (משפחות)

**כפילויות וסדר.** הספק שולח לפעמים אותה חבילה פעמיים, ולפעמים הקריאות בתוך חבילה
לא מסודרות לפי זמן. הפתרון בשתי שכבות: זיהוי מהיר בזיכרון לפני העיבוד, וכתיבה
מותנית בבסיס הנתונים שהיא קו ההגנה הסופי — לא מניחה סדר ולא תספור פעמיים.

**תיקונים מאוחרים.** הכתיבה המותנית דואגת שהערך החדש יחליף את הישן לפי
`received_at`, וטריגר ב-DB מתעד אוטומטית כל תיקון (ערך ישן, חדש, הפרש).

**הפסקות פיד.** כשהספק שותק, לוח הבקרה מתריע ("DEAD") דרך תפוגת מפתח בזיכרון,
ושירות הגיבוי ממלא את הפער מהארכיון ברגע שהשידור חוזר.

**זמן-קיר מול זמן-נתונים.** אחת הבעיות העקרוניות. הסימולציה מריצה נתונים מ-2013–
2014, בעוד שעון המחשב הוא 2026. כל לוגיקה שהסתמכה על `NOW()` חישבה חלון ב-2026
בעוד הנתונים ב-2014, וקיבלה אפס תוצאות. הפתרון: כל מה שתלוי בזמן (החלון החי, זיהוי
מונים שותקים, גרפי ההיסטוריה) מתבסס על *הזמן האחרון בנתונים עצמם* ולא על שעון הקיר.

**עמידוּת ואי-אובדן.** השילוב של checkpoint על דיסק, כתיבה מותנית אידמפוטנטית,
ושחרור סימוני כפילות בכשל כתיבה — מבטיח שאצווה שנכשלה תעובד שוב ולא תיעלם.

### 10.2 הבאגים הספציפיים שתוקנו ואומתו בהרצה חיה

- **התחזית הייתה תקועה על אפס.** חישוב התחזית ערבב שעון-קיר (2026) עם זמן-החלון
  (2014), כך שה"זמן שעבר" יצא כ-12 שנה והתחזית ≈0. החלפנו לגישת **תחזית מבוססת-
  כיסוי** (סעיף 6, שכבה 6) — גם נכון יותר וגם עובד בסביבת הסימולציה.
- **זיהוי מונים שותקים אף פעם לא החזיר תוצאות.** השאילתה השתמשה ב-`NOW()` מול
  נתוני 2014. עיגנו אותה ל-`MAX(reading_time)`, וכעת היא מזהה כראוי.
- **אובדן נתונים שקט בכשל כתיבה.** אם הכתיבה ל-DB נכשלה, המנוע "בלע" את השגיאה
  והמשיך — והכפילות כבר סומנה כ"נראתה", כך שבריצה חוזרת הקריאות נעלמו. תיקנו
  שכשהכתיבה הקריטית נכשלת, המנוע משחרר את סימוני הכפילות וזורק שגיאה → האצווה
  מורצת מחדש בלי לאבד דבר.
- **פאנל הדחיות נשאר ריק.** קובץ נתונים לפי זמן ההוספה (2026) שמחוץ לחלון הלוח.
  הפנינו אותו לקרוא ישירות מטבלת הדחיות ולקבץ לפי זמן-הנתונים.
- **Spark קרס בזיכרון (OOM).** ניסה למשוך מיליוני קריאות לזיכרון בבת אחת, כי
  מגבלת האצווה נספרה ב"הודעות" בעוד כל עמוד גיבוי נושא אלף קריאות. הגבלנו את גודל
  האצווה (`maxOffsetsPerTrigger=500`) והגדלנו את זיכרון ה-driver.
- **שאילתת הגנה מס' 5 נכשלה.** שגיאת טיפוס — `ROUND` אינו תומך ב-double ללא המרה.
  הוספנו cast ל-`numeric`.
- **הביצועים — צוואר הבקבוק האמיתי.** הלוח "קפא" לדקות. החקירה הראתה שהכתיבה
  שורה-שורה איטית מדי: אצווה של ~2M שורות לקחה ~6 דקות. החלפנו ל**COPY לטבלת
  ביניים ואז MERGE יחיד** — מהיר פי עשרות מונים (כעת ~20ש') — וכיווננו `work_mem`.
- **שני באגים שנחשפו בהזרקת נתוני בדיקה.** (א) הולידציה דחתה רק זמן *ריק* אך לא
  זמן *לא-תקין*, כך שזמנים פגומים נשמטו בשקט; (ב) מנוע המצב החי קרס כשנתקל בזמן
  כזה. תיקנו את שניהם.
- **פאנלים שלא הציגו.** "Active Meters" הראה "אין נתונים" כי PFCOUNT לא מחזיר
  data frame לתוסף — המנוע כותב כעת למפתח `window:meters` רגיל. כיילנו מחדש ספי
  צבע והגדרנו "0" במקום "אין נתונים".
- **"Feed Status" לא הציג "LIVE" במצב תקין.** מפתח `window:last_batch` נכתב עם
  TTL 600ש' בעוד הלוח ממפה רק TTL 1–150 ל-LIVE; ה-TTL ריחף סביב 595–600 והפאנל
  הציג מספר גולמי, וגם "DEAD" קפץ רק אחרי 10 דקות. החזרנו את ה-TTL ל-150, כך
  ש-"LIVE" מוצג כשהפיד חי ו-"DEAD" קופץ תוך ~150ש'.
- **פער עמידוּת: אובדן תיקון בהפסקת עיבוד ארוכה.** ה-retention של Kafka היה שעה
  אחת. תיקון מאוחר מגיע אך ורק בזרם החי ו**אינו** בארכיון; אם Spark למטה מעבר
  ל-retention, ההודעה נמחקה והגיבוי החזיר רק את הערך **המקורי** → התיקון אבד.
  **הפתרון:** הארכנו את ה-retention ל-**6 שעות** (מרווח רחב כי Spark קם תוך
  שניות), בתוספת נוהל "DEAD נמשך → restart ל-Spark". נותר קצה-מקרה תיאורטי בלבד
  (ראה ממצא ד.5).
- **דליפת מנויים — כל פרוסה נמסרה ×4.** בבדיקה התגלה שלספק נרשמו **ארבעה מנויים**
  שכולם מצביעים על השגשר (ועוד מנוי מת ל-`example.com`). הסיבה: הספק שומר את
  המנויים לדיסק ומשחזר אותם בכל הפעלה מחדש, כך שלאורך סבבי `vendor-chaos`/
  `vendor-calm` מנויי השגשר הצטברו. ההשפעה: כל פרוסת-ספר חיה נמסרה פי 4, ניפחה את
  אצוות Spark ודחפה את מנוע העיבוד מאחור — וזה היה השורש האמיתי של הצטברות הנתונים
  שנצפתה בקצב הדמו (לא קצב הסטרס). **הפתרון:** הוספנו לשגשר **ניקוי-עצמי אידמפוטנטי**
  בעלייה — לפני ההרשמה הוא מוחק כל מנוי קודם שמצביע עליו, כך שתמיד נשאר מנוי יחיד
  (סעיף 6, שכבה 1). לאחר התיקון העומס החי ירד פי 4 והמערכת עומדת בקצב הדמו (60)
  בלי הצטברות.

## 11. מה עשינו מעבר לדרישות

חלק מהעבודה חרג ממה שנדרש במפורש, מתוך כוונה למסור מערכת ברמת ייצור:

- **מעקב עיכוב שידור לכל מונה** — ממוצע נע, אחוזון 95 ומקסימום, כמדד בריאות עדין
  של הפיד, מעבר ל"חי/מת" הבסיסי. גם ה-`avg` וגם ה-`p95` מוחלקים אקספוננציאלית
  (`*0.8 + new*0.2`) כך שהם יורדים כשהפיד משתפר; ה-`max` נשאר high-water mark (ד.2).
- **זיהוי מונים יתומים** — מונים שמופיעים בקריאות אך לא ברשימת הבתים, לאיתור בעיות
  בנתוני הממד.
- **ארכוב ומחיקה אוטומטיים** — סיכום שעתי קבוע של דחיות לצד מחיקת פרטים אחרי שבוע.
- **תחזית מבוססת-כיסוי** — רעיון שלא היה במפרט; הגענו אליו אחרי שהבנו שהנוסחה
  הנאיבית לא מתאימה לאופי הנתונים.
- **הנדסת ביצועים** — COPY, כיוונון זיכרון, הגבלת גודל אצווה, הגדלת זיכרון המנוע,
  ובסבב זה גם **חלוקת `clean_readings` לפי `reading_time`** ו-**`synchronous_commit=off`**
  בנתיב הכתיבה (סעיף 12.1).
- **עמידוּת מוקפדת** — שחרור סימוני כפילות בכשל, מזהה חבילה יציב לגיבוי, והרצת
  משימות תחזוקה מיד בעלייה עם סובלנות לאיחור.
- **טקסונומיית דחיות** — שלושה סוגי דחייה מובחנים עם תיעוד הסיבה.
- **ליטוש לוח הבקרה** — כיול ספים, "0" במקום "אין נתונים", תיקון חיבור מקורות.
- **כלי דמו** — `tools/inject_demo.py`, עטוף ב-`make demo`, שמזריק קריאות פסולות
  וכפילויות דרך הצינור האמיתי ואז מאמת ש-Grafana חי ושהדאשבורד טעון (12 פאנלים).

## 12. ביצועים וקנה־מידה — ממצא ומסקנות

**הממצא.** בדקנו את המערכת בקצב ברירת המחדל (רפליי 3ש'/יום ≈ 81,000 קריאות
לשנייה) עם הגיבוי פעיל, וגילינו שהיא **נופלת מאחור**: גודל האצוות תפח בהתמדה
(5K → 25K → 38K → 126K → 159K קריאות) וזמן העיבוד חרג מחלון 5 השניות.

**הסיבה.** תקרת קיבולת של מערכת **חד-צמתית**: צוואר הבקבוק הוא קצב הכתיבה ל-DB —
כ-6,000 שורות לשנייה מול טבלה של ~13M שורות — בעוד הקצב הנכנס בברירת המחדל גבוה
פי כ-13. חשוב: זו **אינה רגרסיה** — מעבר ל-COPY שיפר את הקצב פי כמה מונים (אצווה
שלקחה ~6 דקות לוקחת כעת ~20ש').

**מסקנה לגבי הדרישות.** הדרישות הפונקציונליות מתקיימות במלואן — אי-אובדן בהפעלה
מחדש, dedup, תיקונים מאוחרים, שש שאילתות ההגנה, ועדכון הלוח "תוך שניות" בקצב
רפליי סביר. דרישת ה-BRIEF היא דרישת *טריות* (latency), והיא מתקיימת. הקצב 3ש'/יום
מוגדר ב-BRIEF עצמו כ*מהירות סטרס* ("speed it up for stress tests… slow it down
for demos"), ואינו דרישה תפעולית.

**תנוחת דמו יציבה.** לצפייה נוחה: גם השהיית הגיבוי (`docker compose stop
backfill`) **וגם** האטת הסימולציה (`REPLAY_SECONDS_PER_DAY=60` או `300`). השהיית
הגיבוי לבדה **אינה מספיקה** בקצב 3, כי הזרם החי לבדו דוחף פרוסת-ספר שלמה (~5,000
קריאות) בכל מסירה. בקצב מתון האצוות קטנות והלוח טרי ויציב.

### 12.1 מה צריך כדי לעמוד בקצב הסטרס — מה שבוצע ומה שנותר

**הפער.** קצב הסטרס המלא (`REPLAY_SECONDS_PER_DAY=3`) הוא ≈81,000 קריאות לשנייה,
מול נתיב כתיבה חד-צמתי שעמד על ~6,000 שורות/שנייה — פער של פי ~13. חשוב להבחין:
המערכת **לא מאבדת** נתונים בסטרס — היא צוברת אותם **בתור Kafka** (backpressure דרך
`maxOffsetsPerTrigger`), Spark נופל מאחור, והדאשבורד מפגר. הנתונים נכונים בסוף;
מה שעל הקו הוא דרישת ה**טריות** — והיא מתקיימת בקצב מתון. ה-BRIEF עצמו מגדיר את
קצב 3 כ*כפתור סטרס*, לא כדרישה תפעולית.

**מה בוצע בסבב זה (אופטימיזציות צומת-יחיד, מינוף גבוה, ללא תשתית חדשה):**

- **#1 — חלוקת `clean_readings` לפי `reading_time`** (partitions חודשיים + DEFAULT).
  תוקף את צוואר הבקבוק האמיתי — תחזוקת אינדקסים מול טבלה תופחת: כל כתיבה נוגעת רק
  ב-partition הנוכחי הקטן. זול (DDL בלבד). אומת חי: שורות מנותבות נכון ל-partitions.
- **#3 — `synchronous_commit=off` על סשני ה-writer בלבד.** מסיר את עלות ה-fsync
  בנתיב הכתיבה — בטוח כי Kafka הוא היומן העמיד וה-upsert אידמפוטנטי. שאילתות
  האנליסטים שומרות עמידות מלאה.

יחד, 1+3 מעלים משמעותית את תקרת הצומת-היחיד (הערכה ×4–8) ומקטינים את הפער, בלי
מורכבות תפעולית.

**מה נדרש לפי-13 מלא (נדחה, מתועד):**

- **כתיבה מקבילה ב-Spark (`foreachPartition`)** — N executors כותבים במקביל לפי
  `hash(meter_id)`, עם הגדלת מספר ה-Kafka partitions. ×2–4 נוספים, תלוי בליבות.
- **Sharding של PostgreSQL לפי `meter_id`** (Citus/ידני) — הדרך היחידה לפי-13
  אמיתי, אך היקרה ביותר תפעולית. **נדחתה** כעבודה ארכיטקטונית נפרדת ומוגזמת לדמו.
- **ניתוק הנתיב החם** — Redis בקצב מלא לדאשבורד, Postgres אסינכרוני באצוות גדולות,
  כך שהדאשבורד לעולם לא מפגר גם אם נשאר פער כתיבה שיורי.

**החלטה:** ביצענו את 1+3 (צומת-יחיד); קנה-מידה אופקי מלא (sharding) נדחה. תקרת
קצב הסטרס מוכרת, מנותחת ומתועדת.

## 13. החלטות פתוחות וסיכונים שיוריים

בסקירת קוד שורה-שורה עלו ממצאים נוספים — **תצפיות וסיכונים**, לא באגים שוברים.
בסבב זה התקבלו לגביהם החלטות: רובם **תוקנו בקוד ואומתו בהרצה חיה**, היתר התקבלו
כסיכונים שיוריים מודעים.

### 13.1 ממצאים שתוקנו בקוד בסבב זה (ואומתו חי)

- **ד.1 — תיקון בו-זמני (`received_at` זהה) ידרוס כעת.** הכתיבה המותנית
  (`db_writer.py`) הורחבה: מלבד `received_at` מאוחר יותר, היא דורסת גם כאשר
  `received_at` **זהה** (NULL-safe) **וגם** ה-ingest חדש יותר **וגם** ה-`kwh`
  שונה — כך תיקון עם אותו `received_at` מוחל, אך כפילות אמיתית (ערך זהה) נשארת
  no-op. סוגר את הפגיעה ב-"late corrections must overwrite".
- **ד.2 — `p95_lag_sec` עבר להחלקה אקספוננציאלית.** במקום `GREATEST` (שרק עולה),
  העמודה מוחלקת `*0.8 + new*0.2` כמו `avg_lag_sec`, כך שהיא משקפת p95 מתגלגל אמיתי
  שיכול לרדת כשהפיד משתפר. `max_lag_sec` נשאר `GREATEST` (high-water mark נכון).
- **ד.3 — SQL ב-scheduler עבר לפרמטרים מבונדים.** שאילתות השתיקה והניקוי עברו
  מ-`% int(env)` ל-placeholders מבונדים (`%s * INTERVAL '1 hour'` וכו'), לעקביות
  ולניקיון מול שאר השאילתות הפרמטריות.
- **ד.4 — ארכוב דחיות עבר לסכימה מצטברת מדויקת.** במקום `GREATEST`, השאילתה כעת
  סוכמת (`count = existing + new`) ומסמנת כל שורה שנקלטה ב-`archived=TRUE`
  (עמודה חדשה + partial index), ומצרפת רק שורות `NOT archived` — כך כל דחייה נספרת
  בדיוק פעם אחת, בלי תת-ספירה ובלי ספירה כפולה בין ריצות.
- **ד.6 — החלון החי מתגלגל קדימה בלבד.** `_apply_to_window` דוחה כעת אצווה שחלונה
  ישן מהחלון השמור (אצווה טהורת-גיבוי), במקום לגלגל אחורה ולקלקל זמנית את הסכום החי.
- **#1 (ביצועים) — `clean_readings` מחולקת לפי `reading_time`** (partitions חודשיים
  + DEFAULT). מקטין את עלות תחזוקת האינדקסים בכתיבה ומאיץ את שאילתות הטווח (ראה
  סעיף 12.1).
- **#3 (ביצועים) — `synchronous_commit=off` על סשני ה-writer בלבד.** מסיר את עלות
  ה-fsync בנתיב הכתיבה (Kafka הוא היומן העמיד); שאילתות האנליסטים שומרות עמידות מלאה.

### 13.2 החלטות שהתקבלו ללא שינוי קוד (סיכונים שיוריים מודעים)

- **מהירות הרפליי (`.env`): להשאיר `3`, עם כיוונון ידני.** זהו קצב הסטרס; ברירת
  המחדל נשארת 3 וניתן להאיט ידנית לדמו (`REPLAY_SECONDS_PER_DAY=60/300`).
- **קנה-מידה אופקי מלא (sharding): לא להשקיע.** המערכת נשארת חד-צמתית; ביצעו את
  אופטימיזציות הצומת-היחיד (חלוקה + `synchronous_commit`), אך פי-13 מלא דורש
  sharding שנדחה (ראה סעיף 12.1). קצה קצב הסטרס מוכר, מנותח ומתועד.
- **ד.5 (retention): 6 שעות כפתרון סופי.** קצה-המקרה התיאורטי שנותר (הפסקה ארוכה
  מ-6 שעות **יחד** עם תיקון בחלון שנמחק) **מתקבל כסיכון שיורי מודע**. אם אי-פעם
  יידרש: retention ארוך יותר, או נתיב-תיקונים בארכיון עם cursor `corrected_since`.
- **עמידוּת השגשר (אישור לפני שמירה ב-Kafka): להשאיר א-סינכרוני.** יש חלון זעיר
  שבו נפילת השגשר תאבד חבילה מאושרת. שליחה חוסמת בקצב סטרס חוסמת את לולאת האירועים.
  הגיבוי משחזר את כל הקריאות **הרגילות** מהארכיון, כך שהסיכון מצטמצם לתיקונים
  מאוחרים בלבד — אותו שורש כמו ד.5, מתקבל כסיכון שיורי מודע.

## 14. מצב נוכחי, אימות והפעלה

### 14.1 סטטוס

המערכת עומדת בכל שלוש הדרישות: לוח בקרה תפעולי חי שמתעדכן תוך שניות, משטח
אנליסטים עם שש שאילתות ההגנה (כולן רצות), ובסיס תפעולי שעולה בפקודה אחת ושורד
הפעלות מחדש. תיקנו למעלה מעשרה באגים אמיתיים — חלקם עקרוניים — ואימתנו את כולם
בהרצה חיה, כולל הזרקת נתונים שגרמה לכל הפאנלים להציג ולהשתנות על המסך.

### 14.2 שמירת נתוני דמו

1. **אוטומטי** — כל הנתונים ב-Docker volume `lume_pg-data`, ששורד `stop`/`start`/
   `restart`. נמחקים *רק* ב-`make reset`.
2. **גיבוי נייד** — תמונת מצב דחוסה: `data/backups/lume_demo_2026-06-19.dump`
   (~147MB, כל 11 הטבלאות, מוגן מ-git). שחזור:
   ```
   docker compose exec -T postgres pg_restore -U lume -d lume --clean --if-exists \
       < data/backups/lume_demo_2026-06-19.dump
   ```
3. **ייצור על דרישה** — `tools/inject_demo.py` (או `make demo`) מזריק קריאות
   (תקינות ופסולות) דרך הצינור האמיתי.

### 14.3 הפעלה

```
make run            # בונה ומעלה הכול בפקודה אחת
make demo           # מזריק קריאות דמו ומאמת שגרפנה והדאשבורד חיים
make vendor-chaos   # מפעיל כפילויות/ערבול/הפסקות
make vendor-calm    # מכבה chaos
make stop / reset   # עצירה / איפוס (מוחק volumes)
```

**כתובות:** דאשבורד גרפנה — `http://localhost:13000` (Lume → Control Room) ·
Vendor — `http://localhost:18100/docs` · Ingest health — `http://localhost:18080/healthz`.

### 14.4 אימות חי (סבב נוכחי)

הסבב הנוכחי כלל העלאה מלאה של הסטאק (`docker compose up --build` בקצב מתון 60),
לאחר אתחול נקי של מנוע Docker ומחיקת volumes של ה-DB/Kafka/checkpoint כדי שהסכמה
המחולקת החדשה תיכנס לתוקף. כל הבדיקות עברו:

| בדיקה | תוצאה |
|---|---|
| בריאות 9 השירותים | כולם `Up`; kafka/meter-vendor/postgres/redis/ingest = `healthy` |
| **חלוקה (#1)** | 7 partitions נוצרו; שורות מנותבות נכון לפי `reading_time` (2013-12 ו-2014-01 התמלאו בהתאם) |
| **`synchronous_commit=off` (#3)** | כתיבות COPY+MERGE מצליחות ללא שגיאה; אצוות מעובדות |
| **ד.4 (`archived`)** | העמודה קיימת; שאילתת הארכוב המצטברת תקפה |
| ממדים | `homes`=5,083 · `weather_hourly`=17,509 נטענו |
| שש שאילתות ההגנה | כולן רצו `EXIT=0` ללא שגיאה, כולל ה-cast ל-`numeric` בשאלה 5 |
| גרפנה | `HTTP 200`, `database: ok`; דאשבורד "Lume — Control Room" עם **12 פאנלים**; שני מקורות נתונים (PostgreSQL + Redis) מחוברים |
| חלון חי (Redis) | `window:kwh`≈1,788 · `window:meters`=5,030 · `window:forecast`≈1,807 · `window:last_batch` TTL=57 → **Feed Status = LIVE** |
| **תיקון דליפת מנויים** | לפני: 4 מנויי שגשר + מנוי מת ל-`example.com`. אחרי הניקוי-העצמי: **מנוי יחיד**; העומס החי ירד פי ~4 |
| **עמידה בקצב 60 (1×)** | לאחר התיקון גידול האצוות ה"בורח" נעצר (האצוות הגיעו לתקרת ה-500-הודעות וקפאו במקום לתפוח); קצב הייצור החי במצב יציב (~4K קריאות/ש') מתחת לתקרת הכתיבה (~6.3K/ש') → המערכת עומדת בקצב הדמו |

**הערה על אימות הדליפה:** המופע ארוך-החיים שנבדק עדיין נושא backlog שהצטבר בתקופת
ה-×4, והוא מתנקז עם הזמן; הפעלה נקייה (`make run`) מתחילה ממנוי יחיד וללא הצטברות
מלכתחילה. אין אובדן נתונים בשום שלב — Kafka חוצץ את ה-backlog על הדיסק.

**הערה תפעולית מהסבב:** מנוע Docker (WSL backend) נמצא במצב תקוע בתחילת הסבב
(תוצאת שינה/חידוש של הלפטופ) — פעולות קריאה עבדו אך כל `stop`/`exec`/`rm` נתקעו.
הפתרון: `wsl --shutdown` ואז `up --build`. אין לכך קשר לקוד; זו תקלת סביבה ידועה
ב-Docker Desktop ל-Windows.
